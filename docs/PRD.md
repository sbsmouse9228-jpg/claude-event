## 1. 제품 개요

### 1.1 배경 및 문제 정의

수영, 헬스, 친구 소셜 모임 등의 주최자는 매번 다음 작업을 수작업으로 반복한다.

- 카카오톡 단체 채팅방에 공지 발행
- 개인 DM으로 출석 여부 확인
- 카풀 희망자 취합 및 차량 배정
- 모임 후 비용 정산 및 송금 요청 추적

이 과정은 파편화되어 있어 주최자의 리소스를 과도하게 소모하고, 참여자 입장에서도 여러 메시지를 확인해야 하는 불편함이 있다.

### 1.2 솔루션

**MOA(모아)** 는 모임 주최자가 하나의 링크로 공지·참석 확인·카풀·정산을 한 번에 처리할 수 있는 경량 이벤트 관리 웹 플랫폼이다.

### 1.3 핵심 가치 제안

| 대상 | 가치 |
|------|------|
| 주최자 | 반복 업무 자동화 — 공지 1회 발행으로 리마인드·정산 요청까지 처리 |
| 참여자 | 링크 하나로 출석 확인, 카풀 신청, 정산 완료 |

## 2. 타겟 사용자

### 2.1 주요 페르소나

**페르소나 A — 모임 주최자 (김지수, 32세)**
- 매주 토요일 아침 수영 모임을 운영
- 매번 카카오톡으로 인원 확인, 카풀 조율, 정산 요청을 개별 DM으로 처리
- 10명 내외 고정 멤버 + 가끔 게스트 참여
- 불편함: 출석 미확인자 개별 연락, 정산 후 미송금자 추적이 번거로움

**페르소나 B — 일반 참여자 (박민준, 28세)**
- 여러 모임에 참여하는 직장인
- 모임별로 채팅방이 달라 공지를 놓치는 경우가 잦음
- 정산 금액이 얼마인지 매번 물어봐야 함
- 니즈: 간단한 링크 클릭으로 모든 것을 처리하고 싶음

### 2.2 타겟 모임 유형

- 정기 운동 모임 (수영, 헬스, 테니스 등)
- 친구 소셜 모임 (번개, 정기 식사, 여행)
- 동호회 / 클럽 모임

### 2.3 모임 규모

MVP 타겟: **소규모 5~15명**

### 3.1 포함 기능

| 우선순위 | 기능 영역 | 설명 |
|---------|----------|------|
| P0 | 인증 | 카카오 소셜 로그인 |

| P0 | 공지 & 참여자 관리 | 공지 발행, 출석 확인(참가/불참/미정), 리마인드 |
| P0 | 카풀 매칭 | 차량 등록, 탑승 신청, 자동 매칭 |
| P0 | 정산 시스템 | 비용 항목 입력, 더치페이, 개인별 납부 확인 |
| P1 | 알림 | 이벤트 생성·D-1·정산 요청 알림 |

- 모임 내 채팅 기능
- 결제 연동 (토스페이, 카카오페이 직접 송금)
- 모임 공개 검색 / 신규 멤버 모집

## 4. 기능 상세 요구사항

### 4.1 인증

```
FR-AUTH-01: 카카오 OAuth 2.0 로그인
FR-AUTH-02: 로그인 없이 초대 링크(/invite/[token])로 이벤트 조회 가능
FR-AUTH-03: 비회원은 카카오 로그인 후 자동으로 원래 이벤트로 리다이렉트
```

### 4.2 이벤트 생성 및 관리

```
FR-EVENT-01: 주최자는 이벤트를 생성할 수 있다
  - 필수 입력: 제목, 날짜/시간, 장소
  - 선택 입력: 최대 인원, 설명, 참가비

FR-EVENT-02: 이벤트 생성 시 고유 초대 링크(토큰)가 자동 발급된다
FR-EVENT-03: 주최자는 초대 링크를 복사하여 카카오톡 등으로 공유할 수 있다
FR-EVENT-04: 주최자는 이벤트 정보를 수정·삭제할 수 있다
FR-EVENT-05: 이벤트 상태는 [예정 / 진행중 / 완료 / 취소] 4가지이다
```

### 4.3 공지 & 참여자 관리

```
FR-NOTICE-01: 주최자는 공지사항을 작성하여 참여자에게 발송할 수 있다
FR-NOTICE-02: 공지 발송 시 참여자에게 알림이 전송된다

FR-ATTEND-01: 참여자는 출석 상태를 선택할 수 있다 (참가 / 불참 / 미정)
FR-ATTEND-02: 주최자는 실시간 출석 현황을 확인할 수 있다 (N/전체 명 참가)
FR-ATTEND-03: 주최자는 미응답 참여자에게 리마인드 알림을 일괄 발송할 수 있다
FR-ATTEND-04: 최대 인원 초과 시 추가 참가 신청이 불가하다
```

### 4.4 카풀 매칭

```
FR-CARPOOL-01: 참여자는 차량을 등록할 수 있다
  - 입력: 출발 위치, 출발 시간, 탑승 가능 인원

FR-CARPOOL-02: 참여자는 탑승을 신청할 수 있다
  - 등록된 차량 목록 확인 후 신청

FR-CARPOOL-03: 차량 제공자는 탑승 신청자를 수락/거절할 수 있다
FR-CARPOOL-04: 수락 시 해당 참여자에게 차량 정보(제공자 연락처, 출발 장소)가 전달된다
FR-CARPOOL-05: 카풀 현황 페이지에서 전체 카풀 그룹 구성을 확인할 수 있다
```

### 4.5 정산 시스템

```
FR-SETTLE-01: 주최자는 비용 항목을 입력할 수 있다
  - 항목명, 금액, 부담 대상 (전원 / 선택 멤버)

FR-SETTLE-02: 시스템은 참여자별 납부 금액을 자동 계산한다 (더치페이 기본)
FR-SETTLE-03: 주최자의 계좌 정보를 등록하면 정산 페이지에 표시된다
FR-SETTLE-04: 참여자는 송금 완료 후 '납부 완료' 버튼으로 확인 처리한다
FR-SETTLE-05: 주최자는 미납 참여자에게 알림을 발송할 수 있다
FR-SETTLE-06: 정산 완료율이 표시된다 (N/M명 완료)
```

### 4.6 알림

```
FR-NOTIFY-01: 이벤트 생성 → 초대 링크 포함 알림 (주최자 수동 공유)
FR-NOTIFY-02: 이벤트 D-1 → 미출석 확인자 자동 리마인드
FR-NOTIFY-03: 카풀 수락 → 탑승 신청자에게 차량 정보 알림
FR-NOTIFY-04: 정산 등록 → 참여자 개인별 납부 금액 알림
FR-NOTIFY-05: 알림 채널: 웹 푸시 (MVP) / 카카오 알림톡 (v2)
```

## 5. 비기능 요구사항

| 항목 | 요구사항 |
|------|---------|
| 반응형 | 모바일 퍼스트 (375px 기준 최적화) |
| 성능 | 페이지 로드 3초 이내 (LCP 기준) |
| 보안 | 초대 링크 토큰은 UUID v4, 만료 없음 (이벤트 삭제 시 무효화) |
| 접근성 | 비회원도 이벤트 조회 가능 (출석 확인은 로그인 필요) |
| 데이터 | 이벤트 완료 후 1년간 히스토리 보관 |

## 6. 기술 스택

| 영역 | 기술 | 선택 이유 |
|------|------|---------|
| Frontend | Next.js 14 (App Router) + TypeScript | SSR, 모바일 최적화 |

| 알림 (MVP) | Web Push API | 별도 비용 없음 |

## 7. 데이터 모델 (개요)

```
users
  ├─ id (uuid)
  ├─ kakao_id
  ├─ nickname
  └─ profile_image

events
  ├─ id (uuid)
  ├─ host_id → users.id
  ├─ title, description
  ├─ location, event_date
  ├─ max_participants
  ├─ invite_token (uuid)
  └─ status (scheduled | ongoing | completed | cancelled)

event_participants
  ├─ event_id → events.id
  ├─ user_id → users.id
  └─ attendance (attending | absent | pending)

carpool_cars
  ├─ event_id → events.id
  ├─ driver_id → users.id
  ├─ departure_location
  ├─ departure_time
  └─ available_seats

carpool_requests
  ├─ car_id → carpool_cars.id
  ├─ passenger_id → users.id
  └─ status (pending | accepted | rejected)

settlements
  ├─ event_id → events.id
  ├─ item_name, amount
  └─ split_type (equal | custom)

settlement_payments
  ├─ settlement_id → settlements.id
  ├─ user_id → users.id
  ├─ amount_due
  └─ paid_at (null이면 미납)

notices
  ├─ event_id → events.id
  ├─ author_id → users.id
  └─ content, created_at
```

## 8. 화면 구성 (사이트맵)

```
/                           랜딩 페이지 (서비스 소개 + 카카오 로그인)
/dashboard                  내 모임 목록 (주최 / 참여)
/event/create               이벤트 생성 폼
/event/[id]                 이벤트 메인 (참여자 뷰)
  ├─ 이벤트 정보 + 공지
  ├─ 출석 확인 버튼
  ├─ 카풀 현황 요약
  └─ 정산 현황 요약
/event/[id]/manage          주최자 관리 패널
  ├─ 참여자 현황 + 리마인드
  ├─ 공지 작성
  └─ 이벤트 수정
/event/[id]/carpool         카풀 상세 (차량 등록 / 탑승 신청)
/event/[id]/settle          정산 상세 (비용 입력 / 납부 확인)
/invite/[token]             초대 링크 진입점 (비회원 접근 가능)
```

## 9. 개발 로드맵

| 단계 | 기간 | 내용 | 완료 기준 |
|------|------|------|---------|
| **1단계** | 1~2주차 | 프로젝트 세팅, 카카오 로그인, 이벤트 생성, 초대 링크 | 링크 공유 후 비회원이 이벤트 조회 가능 |
| **2단계** | 2~3주차 | 출석 확인, 공지 발행, 참여자 대시보드 | 주최자가 출석 현황 실시간 확인 가능 |
| **3단계** | 3~4주차 | 카풀 등록, 탑승 신청, 매칭 | 카풀 그룹 자동 구성 및 알림 |
| **4단계** | 4~5주차 | 정산 입력, 더치페이, 납부 확인 | 미납자 알림 발송 가능 |
| **5단계** | 5~6주차 | 통합 테스트, 버그 수정, Vercel 배포 | 실제 모임에서 테스트 완료 |

## 10. 성공 지표 (MVP)

| 지표 | 목표 |
|------|------|
| 이벤트 생성 수 | 베타 기간 내 30개 이상 |
| 초대 링크 클릭률 | 발송 대비 70% 이상 |
| 출석 확인 완료율 | 이벤트당 참여자 80% 이상 |
| 정산 완료율 | 정산 등록 이벤트의 90% 이상 납부 확인 |
| 재사용률 | 주최자 2회 이상 이벤트 생성 비율 50% |
